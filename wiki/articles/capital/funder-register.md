## Operating Loop

Use this sequence so the table becomes an actionable pipeline rather than a static list.

| Step | What happens | System of record | Human gate |
|---|---|---|---|
| 1. Find | CivicScope / Grantscope surfaces funders, grants, buyers and procurement targets. | CivicScope Goods workspace, Goods grant feed and Goods admin pipeline. | Check the opportunity is real, current and relevant to Goods. |
| 2. Classify | Mark the target as received, applied, warm, prospect or research. Separate cash, receivables, pipeline and demand signals. | This register plus `/admin/pipeline`. | Do not let pipeline amounts become received funding. |
| 3. Match evidence | Attach the right proof: product proof, community voice, production pathway, finance need, governance note, buyer signal or impact evidence. | Wiki source pages and Notion review queue. | Ben / Nic review what can be shared. |
| 4. Decide next action | Decide whether this is a funder action, procurement action, capital action or partner action. | Notion action list and Goods admin dashboards. | Only approved targets move to CRM action. |
| 5. Push to GHL | Approved targets become GHL contacts, notes and opportunities with tags like `goods-capital-target`, `goods-buyer-target` and `goods-partner-target`. | GoHighLevel plus Supabase `crm_contacts` / `crm_deals`. | Requires human approval before push because it creates or changes CRM records. |
| 6. Follow up | GHL carries the next touch, owner, stage and reminders. | GHL pipeline and `/admin/network`. | Keep notes short and point back to the evidence page. |
| 7. Reconcile | When money, an order or a formal application changes status, update this page and the finance/pipeline records. | Wiki, Supabase CRM and finance records. | Reconcile before using in mentor or investor materials. |

## GHL Push Rules

The existing technical path is already in the app: approved outreach targets can be pushed through `/api/admin/targets/push-outreach`, which maps targets into GHL and Supabase CRM.

Use it carefully:

- Start from the Goods admin review surfaces, not directly from a raw CivicScope API response.
- Use dry run first so the team can see the contact, tags, note and opportunity before a CRM record is changed.
- Push capital targets with `goods-capital-target`.
- Push buyer and procurement targets with `goods-buyer-target`.
- Push community, distribution and manufacturing partners with `goods-partner-target`.
- Do not push raw story material, household data, private notes, speculative amounts or unreviewed contact details.
- Do not push a target only because it appears in Grantscope. It needs a clear next action and owner.
