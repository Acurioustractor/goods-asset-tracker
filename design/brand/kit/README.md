# Goods on Country — Claude Design kit

A small studio for building many on-brand artifacts (decks, one-pagers, stat cards,
social tiles, data-room pages) fast, all from one source. Everything is HTML so it
stays version-controlled, regenerable, and exportable to PDF or Adobe Express.

## What's here

| File | What it is |
|---|---|
| `kit.css` | Shared brand components (tokens, type, stat cards, quotes, charts, split, dark). Link it; change it once, every artifact follows. |
| `_template.html` | Copy this to start a new artifact. Links `kit.css` + the brand fonts and carries the export metadata. |
| `photo-contact-sheet.html` | Every approved/canon photo in one page, so you pick by eye (not by guessing filenames). |
| `canon-build.html` | **The master review surface.** Every slot the raise needs (filled/seed/empty) on top, then every candidate (repo + EL + video) with a pick id. Generated by `canon-build.mjs`. |
| `el-photo-review.html` | Every Empathy Ledger image + portrait, by shoot date, cached locally. Generated by `el-photo-review.mjs`. |
| `render.sh` | `./render.sh <artifact.html>` → PDF + preview PNG, with all images resolving. |

## The live board is the primary surface
Day-to-day tagging/alignment now happens in the running app at **`/admin/canon`** (not these static
HTML sheets). It shows every EL asset (photos + videos + storyteller portraits) and every repo image,
lets you pin the best one to a **QBE area** or a **purpose slot**, carries consent badges, and saves
instantly to committed JSON. Every pin regenerates `design/canon-resolved.json` (the flat
slot → winner map artifacts read), so the loop closes automatically. The static sheets below are the
offline/print companion and feed the same canon files.

Run order to refresh the board's EL data cache + the offline sheets: `cd v2 && node scripts/el-photo-review.mjs && node scripts/canon-build.mjs`. Collapse picks to the resolved map any time with `node scripts/canon-resolve.mjs` (the board also does this on every pin).

## Wire picks straight into an artifact (the closed loop)
Reference any canon image by its **slot**, not a file path: write `CANON:<slot-key>` inside a
`src="…"` or `url('…')`. When you render, the current winner from `design/canon-resolved.json`
is baked in automatically.

```html
<div class="op-hero" style="background-image:url('CANON:cover-hero')"></div>
<img src="CANON:storyteller-mykel">
```
- `design/brand/kit/render.sh <artifact>` detects `CANON:` tokens and bakes them before rendering.
- Or bake manually: `node v2/scripts/canon-render.mjs <artifact.html> [--inline]` → writes `<artifact>.resolved.html`.
- Resolution per slot: local canon pick → EL pick (from the local `.el-cache`, so it works even when EL is down) → seed. EL CDN downloads are cached in `design/.canon-cache/`.
- `--inline` base64-embeds every image for a self-contained file (Adobe Express / portable PDF).

So: pick on `/admin/canon` → `canon-resolved.json` updates → re-render the deck/one-pager → your pick is in it. `funder-onepager.html` and `claude-design/invest-deck-full.html` are wired this way.

## Numbers the same way (never retype a canon figure)
Reference any canonical figure by its **canon id**: write `CANON:num:<canon-id>` where the
figure goes. The same bake resolves it to the exact formatted string from
`design/canon-numbers.json` (`496`, `2,660kg`, `$713,827`), which is GENERATED from
`v2/src/lib/data/canon.ts` (the apex registry) by `v2/scripts/canon-numbers.mjs`.

```html
<div class="stat"><div class="num">CANON:num:beds-deployed</div><div class="lbl">beds deployed</div></div>
<span class="pill">CANON:num:plastic-kg plastic diverted</span>
```
- When a figure changes: change `canon.ts` first (the standing rule), then `cd v2 && node scripts/canon-numbers.mjs` to regenerate the JSON, then re-render. Never hand-edit `canon-numbers.json`.
- `render.sh` runs `canon-numbers.mjs --check` before every bake that uses number tokens and REFUSES to render if the JSON has drifted from `canon.ts`, so a stale figure cannot ship silently.
- Prefix trick for currency variants: the token bakes `$750`, so `AUCANON:num:stretch-price` bakes to `AU$750`.
- Only figures with a canon id get tokens. Narrative numbers that are not canon facts (the $400K ask, the BOM's $344 leg, break-even bed counts) stay hand-written where they are.
- The deck and `funder-onepager.html` are wired this way; `funder-landscape-onepager.html` carries only pipeline-target amounts (no canon ids), so it has no number tokens.

## Stages the same way (never retype the pipeline)
The internal landscape one-pager's status pills come from GHL, not from hand-typing.
Each pill is a token, `GHL:stage:<funder-key>` (keys: `sefa`, `snow`, `centrecorp`,
`white-box`, `minderoo`, `vfff`, `qbe`), plus one `GHL:stages-asat` stamp line.
`v2/scripts/landscape-stages.mjs` resolves them at render time.

```bash
design/brand/kit/render.sh --live-stages design/brand/kit/funder-landscape-onepager.html
```
- `--live-stages` pulls the funder pipeline live (GET only, via `ghl-people-pull.mjs`
  run from `v2/` so `.env.local` resolves), matches each row by org name through the
  explicit alias map in the script, and saves the result to
  `design/brand/kit/landscape-stages.json` (no GHL contact ids, safe to commit).
- Without the flag, the last saved pull is baked and the stamp says so with THAT
  pull's date. No saved pull yet: every pill reads STALE. Never a silent lie.
- Stage words are GHL's own (Identified, Qualified, Cultivating, Ask made,
  Delivering, Stewarding, Renewing). An unknown GHL stage renders raw, never guessed.
  A funder in the HTML but missing from GHL renders "not in GHL".
- New funder row: add the row in the HTML with a `GHL:stage:<key>` pill AND the key's
  org aliases to `FUNDERS` in `landscape-stages.mjs`. Stage moves belong in GHL
  (`ghl-people-move.mjs`, dry-run first); this pipeline only reads.

## The canon photo/video loop (review absolutely everything → pick → overwrite)
The investment imagery is driven by **purpose slots**, not loose filenames. Each slot is one job an asset must do (cover, product hero, the plant, a named storyteller, a video).

1. **Refresh the pool** — `cd v2 && node scripts/el-photo-review.mjs` (pulls + caches every EL image/portrait) then `node scripts/canon-build.mjs` (builds the master sheet across EL + repo + video).
2. **Review** — open `canon-build.html`. The slots board shows green = canon set, gold = seed (not yet confirmed), red = empty. Below it, every candidate has a pick id (`R#` repo image, `E#` EL photo, `EP#` EL portrait, `EV#` EL video, `RV#` repo video).
3. **Assign / overwrite** — `node scripts/canon-assign.mjs E14=cover-hero EP7=storyteller-mykel RV12=video-build`. Repo picks are referenced in place; EL picks are downloaded once into `v2/public/images/el/`. Each assign overwrites the slot's prior pick and tags the entry with its slot key in `image-canon.json`.
4. **Artifacts follow** — every deck/one-pager/card pulls the asset for its slot, so a swap propagates everywhere.

- The slot spec is `design/canon-slots.json` (add a slot there to expand what the raise needs).
- RED slots (storytellers, identifiable people) are consent-gated: `canon-assign` only fills them because you named the pick, and flags them. You are the consent gate.

## The single sources everything pulls from
- **Brand** → `design/brand/tokens.css` (mirrored into `kit.css`)
- **Photos** → `design/image-canon.json` (the 24 approved images) + `v2/public/images/people/` (cleared portraits)
- **Numbers** → `v2/src/lib/data/canon.ts`
- **Quotes** → `wiki/outputs/2026-06-28-funder-system/cleared-quotes-library.md`

## The loop (how we go back and forth)
1. **Say what you want** — "a one-pager for SEFA", "a stat card for socials", "a data-room cover".
2. **I assemble it** from `kit.css` blocks + the canon photo you point to + canon numbers. On-brand by construction.
3. **`./render.sh <file>`** → you review the PDF/PNG in seconds.
4. **You tweak** — "that photo, tighter headline, sage not terracotta" — I edit, re-render.
5. **Ship it** — PDF straight from the HTML, or export to Adobe Express for drag-edit
   (build inlines `kit.css` + images into a self-contained file first).

## Starting a new artifact
```bash
cp design/brand/kit/_template.html design/brand/kit/my-artifact.html
# pick a size class on the root: .slide (16:9) | .onepager (A4) | .square (1080)
# drop in blocks from kit.css, point images at a canon path, add canon numbers
design/brand/kit/render.sh design/brand/kit/my-artifact.html
```

## Picking photos
Open `photo-contact-sheet.html` (render it, or it's a normal web page). Each tile shows
the subject, type, consent class, and the **canon path** to paste into your artifact.
RED tiles are people — only the consent-cleared faces are in the approved set; never use a
person shot that isn't on the cleared list.

## Rules baked in
- Reuse `kit.css` classes; don't re-style per artifact (keeps everything one design).
- Photos only from the canon set / cleared portraits.
- Numbers only from canon; never retype a figure.
- Voice: zero em dashes, straight quotes, "On Country" capitalised, units no-space (20kg),
  the scabies→RHD pathway is the *why*, never a claimed Goods health outcome.
